# Mongoyia IM

Python WebSocket service for user-to-seller customer service chat.

## Local Windows

```powershell
pip install -r requirements.txt
Copy-Item .env.example .env
.\scripts\start-im.ps1
.\scripts\status-im.ps1 -Healthcheck
.\scripts\stop-im.ps1 -Force
```

Logs are written to `logs/im.out.log` and `logs/im.err.log`. The PID file is `run/im.pid`.

## Linux Test Server

```bash
pip3 install -r requirements.txt
cp .env.example .env
python3 main.py
```

For long-running test servers, use either:

- `deploy/mongoyia-im.service.example` with systemd.
- `deploy/supervisor-mongoyia-im.conf.example` with Supervisor.

After changing `.env` or applying chat-table migrations, restart the IM service.

## Auth Token

Set the same `IM_AUTH_SECRET` in the PHP project `.env` and this IM `.env`.

When `IM_AUTH_SECRET` is set, the initial WebSocket message must include `auth_token`. PHP renders or issues these tokens for:

- `/mall/chat/index?gid=<product_id>` users through `/mall/chat/token`.
- `/backend/mall/kf/index` seller and platform customer-service operators.

If `IM_AUTH_SECRET` is empty, IM keeps the legacy unsigned handshake for handover compatibility. Do not leave it empty on a test server or production-like environment.

## Message Payload Limits

IM validates message payloads before saving or broadcasting:

- `IM_MAX_TEXT_MESSAGE_LENGTH` controls the maximum trimmed text-message length. Default: `2000`.
- `IM_MAX_IMAGE_MESSAGE_LENGTH` controls the maximum image URL length. Default: `2048`.
- Text messages must be non-empty.
- Image messages must use a local `/attachment/` URL generated by the mall upload endpoint. Script URLs, remote URLs, path traversal, and malformed paths are rejected.

## Health Check

```bash
python scripts/im-healthcheck.py --url ws://127.0.0.1:8767
python scripts/im-regression.py --url ws://127.0.0.1:8767 --merchant-uid 37 --product-id 102 --store-id 9
python scripts/im-concurrency.py --url ws://127.0.0.1:8767 --merchant-uid 37 --product-id 102 --store-id 9 --users 5
```

The health check opens a WebSocket connection, registers a temporary user, and requests `chat_list`, which verifies both the WebSocket service and database access.

The regression script opens user and merchant WebSocket clients, sends a user message, receives it on the merchant side, sends a merchant reply, verifies user receipt, checks chat history, verifies the merchant chat list includes the session, and checks auth, scope, and payload rejection paths.

The concurrency script opens several user clients against one merchant, sends messages at the same time, verifies merchant receipt, user-side history, and merchant chat list visibility. It is a lightweight test-server acceptance check, not a production load test.
